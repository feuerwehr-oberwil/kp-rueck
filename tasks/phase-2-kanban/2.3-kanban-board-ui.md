## Technical Notes

### Pragmatic Drag and Drop Implementation

**Why Pragmatic DnD over @dnd-kit?**
1. Better performance for complex scenarios
2. Native browser APIs with progressive enhancement
3. Simpler API for our use case
4. Better edge detection for insertion points
5. Official Atlassian product (well-maintained)

**Key Patterns Used**:
```typescript
// Draggable setup
draggable({
  element,
  getInitialData: () => ({ type: "operation", operation }),
  onDragStart: () => setIsDragging(true),
  onDrop: () => setIsDragging(false),
})

// Droppable setup
dropTargetForElements({
  element,
  canDrop: ({ source }) => source.data.type === "operation",
  getData: ({ input }) => attachClosestEdge(data, { element, input }),
  onDrop: () => { /* handle drop */ },
})

// Monitor for global drag events
monitorForElements({
  onDragStart({ source }) { /* track what's dragging */ },
  onDrop({ source, location }) { /* handle all drops */ },
})
```

### State Management Pattern

**Operations Context**:
- Single source of truth for operations, personnel, materials
- API client handles all backend communication
- Optimistic updates for responsive UX
- Debounced API calls (500ms) to reduce load
- Auto-syncs status changes between related entities

**Data Flow**:
1. User drags person → operation
2. Frontend immediately updates UI (optimistic)
3. Context calls API to persist change
4. If API fails, could roll back (not currently implemented)
5. Next refresh will show server state

### Performance Optimizations

- Client-side only rendering (`isMounted` guard)
- Debounced search and autocomplete (500ms)
- Debounced operation updates (500ms)
- Memoized callbacks where needed
- Filtered data computed on-demand
- Grouped personnel/materials for efficient rendering

### Known Limitations

1. **No real-time sync**: Data loaded once on mount, updates are manual refresh
   - Future: WebSockets or polling for live updates
2. **No conflict resolution**: Concurrent edits could overwrite each other
   - Future: Implement optimistic locking with version numbers
3. **No undo/redo**: Changes are immediate and permanent
4. **No offline support**: Requires active backend connection
5. **Training flag not implemented**: Backend schema has training_flag but not exposed in UI yet
