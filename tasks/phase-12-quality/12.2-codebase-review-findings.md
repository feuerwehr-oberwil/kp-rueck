## 1. Performance Bottlenecks

### 1.1 Backend Performance Issues

#### **CRITICAL: N+1 Query in `get_incidents()` - Status Transitions**

**Location**: `backend/app/crud/incidents.py:52-67`

**Issue**: The function fetches incidents in bulk but then queries status transitions for each incident individually in a loop.

```python
# Current implementation (lines 48-67)
result = await db.execute(query)
incidents = list(result.scalars().all())

# N+1 PROBLEM: Loop through each incident
for incident in incidents:
    # Separate query for EACH incident's status transition
    latest_transition_query = (
        select(StatusTransition.timestamp)
        .where(StatusTransition.incident_id == incident.id)
        .order_by(StatusTransition.timestamp.desc())
        .limit(1)
    )
    transition_result = await db.execute(latest_transition_query)
    latest_timestamp = transition_result.scalar_one_or_none()

    incident.status_changed_at = latest_timestamp if latest_timestamp else incident.created_at

    # Another N+1 query for assigned vehicles
    incident.assigned_vehicles = await _get_assigned_vehicles(db, incident.id)
```

**Impact**:
- For 50 incidents: **100+ database queries** (1 for incidents + 50 for status transitions + 50 for vehicles)
- Estimated overhead: ~500-1000ms on typical loads
- Compounds with polling (every 3 seconds on frontend)

**Recommendation**: Use `selectinload` or subquery with window functions to fetch status transitions in a single query.

#### **MEDIUM: Polling Frequency**

**Location**: `frontend/lib/contexts/operations-context.tsx:362-367`

**Issue**: Frontend polls backend every 3 seconds for full data refresh.

```typescript
// Poll for updates every 3 seconds
const pollInterval = setInterval(() => {
  if (!isLoading) {
    loadData()  // Full data reload
  }
}, 3000)
```

**Impact**:
- ~1200 API requests per hour per user
- Full data reload on every poll (incidents, personnel, materials, settings)
- With N+1 queries, compounds the performance issue

**Recommendation**:

- Implement incremental updates (only fetch changed data)
- Add last_modified timestamps to reduce unnecessary data transfer

**Location**: `frontend/lib/contexts/operations-context.tsx:181-208`

**Issue**: After fetching incidents, the code loops through each incident to fetch assignments.

```typescript
// Lines 165-169: Fetch incidents
const [apiIncidents, apiPersonnel, apiMats, settings] = await Promise.all([...])

await Promise.all(
```

## 3. Missing Error Handling

**Good examples found:**

- `backend/app/api/events.py:212-223` - Catches ValueError for archive validation
- `backend/app/api/assignments.py:47-48` - Catches ValueError for assignment conflicts

**Missing error handling:**
- No database transaction rollback on errors in some CRUD operations
- Limited HTTP status code variety (mostly 404, some 409, few 400)
- No request validation errors logged

**Findings from grep:**
- 9 API files use try/catch blocks ✅
- Error handling is present but could be more consistent

**Recommendation**:
- Standardize error handling with custom exception classes
- Add middleware for global error handling
- Implement proper logging for all errors
- Ensure all database operations are wrapped in transactions

---

#### **MEDIUM: Missing Database Rollbacks**

**Location**: Multiple CRUD functions

**Issue**: Some update operations don't explicitly rollback on error.

**Example** (`backend/app/crud.py:43-54`):
```python
async def update_incident(...):
    db_incident = await get_incident(db, incident_id)
    if db_incident:
        update_data = incident.model_dump(exclude_unset=True)
        for key, value in update_data.items():
            setattr(db_incident, key, value)
        await db.flush()  # No error handling
        await db.refresh(db_incident)
    return db_incident
```

**Recommendation**: Wrap in try/except blocks with explicit rollback.

## 4. Mobile Responsiveness Gaps

- Desktop-first design with sidebars (lines 821-862, 890-931)

**Issues:**
